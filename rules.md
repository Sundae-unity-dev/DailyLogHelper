# 일일회고 정리 규칙

학생이 던진 자유 메모 / 대화 답변을 아래 JSON 스키마로 정리한다.
PDF는 엘리스 트랙 "과제 제출 템플릿" (노션 양식)을 그대로 미러링한다.

## JSON 스키마

```json
{
  "date": "YYYY-MM-DD",
  "student_name": "홍길동",
  "project_order": "2차",
  "filename": "251113_2차 프로젝트_엘리스 트랙_홍길동.pdf",
  "scrum": {
    "today": "오늘 진행한 작업을 한 단락으로 풀어쓴 내용",
    "in_progress": "진행중 작업 단락",
    "tomorrow": "내일 계획 단락"
  },
  "project": "오늘 본인이 기여한 부분. 한 단락 이상으로 살을 붙여 서술. 마크다운 가능 (불릿, 코드, 이미지)",
  "trouble_shooting": {
    "situation": "문제 상황을 한 단락으로 묘사",
    "condition": "재현 조건을 한 단락으로 묘사",
    "resolution": "어떻게 검증·해결했는지 한 단락으로 묘사"
  },
  "links": {
    "github": "https://github.com/...",
    "blog": "",
    "velog": ""
  },
  "study_links": [
    { "title": "부트스트랩 설치방법", "url": "https://..." },
    "https://docs.unity3d.com/..."
  ]
}
```

## 필드별 정리 톤

### date / student_name / project_order / filename
- `date`는 학생이 말한 날짜 그대로. 안 줬으면 오늘 날짜 자동.
- `student_name`은 처음 1회만 묻고 다음부턴 자동 재사용.
- `project_order`는 매 실행마다 묻는다. 학생이 답한 값을 그대로 저장하되, **뒤에 " 프로젝트"가 자동으로 붙는 형태**로 정규화:
  - `2` / `2차` / `이차` → `"2차"`
  - `최종` / `최종 프로젝트` → `"최종"`
  - `미니` / `미니프로젝트` → `"미니"`
  - 즉, JSON의 `project_order` 값 + " 프로젝트" 조합이 자연스러운 한국어가 되도록 저장.
- `filename`은 `yymmdd_{project_order} 프로젝트_엘리스 트랙_OOO.pdf` 규칙으로 자동 생성.
  학생이 명시적으로 다른 이름을 요청하면 그걸 사용.

### 전반적 톤 (★ 중요)
- 모든 본문 필드(scrum 3종 / project / trouble_shooting 3종)는 **한 단락(2~4줄) 정도로 살을 붙여** 작성한다.
- 학생이 던진 짧은 메모를 그대로 옮기지 말고, **사실 범위 안에서 맥락·동기·결과를 자연스럽게 풀어쓴다.**
  - 예: 학생 메모 "유니티 세팅" → PDF "오늘은 본격적인 개발을 시작하기 전 단계로 유니티 프로젝트 세팅을 진행했다. 패키지 임포트와 폴더 구조 정리를 우선적으로 마치고…"
- **메모에 없는 내용을 지어내지 말 것.** 작업 결과·수치·기술스택 등 구체사실은 학생이 명시한 경우에만 적는다.
- 문어체 또는 체언형 어느 쪽이든 자연스러운 쪽으로. 학생 톤이 캐주얼하면 그 톤을 유지해도 OK.
- 불릿이 더 가독성 높은 경우(목록 나열, 체크리스트)는 불릿 사용 OK. 하지만 한 줄짜리 불릿만 던지지 말고 항목마다 짧은 설명을 곁들인다.

### scrum
- 학생이 "스크럼 안 했어요" / "오늘 스크럼 없음"이라고 하면 세 필드 모두 빈 문자열.
- 메모가 두서없으면 다음 순으로 분류:
  - **today** → 오늘 새로 시작하거나 끝낸 작업
  - **in_progress** → 어제부터 이어지는, 아직 안 끝난 작업
  - **tomorrow** → 내일 할 계획 / 목표
- 각 필드는 한 단락. 너무 빈약하면 학생 메모의 키워드를 토대로 "왜 그 작업을 했는지" 한 줄 정도 맥락 보강.

### project (가장 중요)
- 학생이 "오늘 ~ 만들었어요" 라고 던진 메모를 토대로 **본인이 기여한 부분**만 추림.
- **최소 한 단락 이상.** 무엇을(작업) / 어떻게(방법·툴) / 왜(목적·이유) 중 학생이 말한 만큼만 자연스럽게 풀어쓴다.
- 마크다운 자유롭게 사용: 불릿 / 굵게 / 코드블록 / 이미지(`![](경로)`).
- 작업 항목이 여러 개면 본문 한 단락 + 불릿 목록 조합도 좋음.
- 짧은 코드는 ``` 코드블록 ```으로.

### trouble_shooting
- 학생이 "오늘 트러블 없었어요"라고 하면 세 필드 모두 빈 문자열.
- 메모를 다음 3단계로 강제 분해 (각 단계 한 단락):
  1. **situation**: 어떤 증상이었나 (에러 메시지, 비정상 동작 묘사). 단순 "에러 났음"이 아니라 "어디서, 어떤 동작 중에" 까지 풀어쓴다.
  2. **condition**: 어떤 조건에서 재현되었나 (특정 입력 / 환경 / 빌드 설정 / 팀 합의 과정 등). 의견 충돌·기획 트러블도 여기 해당.
  3. **resolution**: 어떻게 검증하고 어떻게 고쳤나 (검색 키워드, 적용한 코드, 검증 방법, 합의된 결론). "어떻게 결정에 이르렀는지" 과정을 한 단락으로.
- 메모가 두서없어도 이 순서로 재배열. 한 단계가 비면 짧은 추정 한 줄로 채우거나 그대로 빈 문자열.

### links / study_links
- URL이 보이면 자동 추출.
- 깃헙/블로그/벨로그는 도메인으로 분류 (`github.com` → github 필드 등).
- 기술 학습 링크는 `study_links` 배열. 학생이 제목을 같이 적었으면 `{title, url}` 객체로, 아니면 그냥 URL 문자열로.

## 이미지 첨부 처리 (★ 중요)

학생이 메모 중간에 스크린샷/이미지 경로를 던지는 경우(`C:\Users\...png`, `~/Pictures/...jpg`, 드래그앤드롭으로 받은 절대경로 등), **메모 맥락을 보고 자동으로 적절한 섹션에 라우팅**한다.

라우팅 규칙(맥락 우선, 키워드는 참고):

| 이미지 종류 / 키워드 | 들어갈 섹션 |
|---|---|
| 회의 화이트보드, 스크럼 보드, 일정표, 칸반 캡처 | `scrum.today` 또는 `scrum.in_progress` |
| 게임 플레이 화면, 에디터 스크린샷, 인스펙터, 구현 결과물, UI 스케치, 기획서 | `project` |
| 빨간 에러 메시지, 콘솔 로그, 익셉션 스택트레이스 | `trouble_shooting.situation` |
| 환경설정 창, 빌드 세팅, 패키지 매니저 캡처 (트러블 맥락) | `trouble_shooting.condition` |
| 수정 후 정상 화면, 비교 전후, 합의된 기획안 캡처 | `trouble_shooting.resolution` |

- 모든 이미지는 `![설명](경로)` 형태로 변환해 **본문 단락 끝에 자연스럽게 삽입.** 단락 중간 끼워넣기 금지.
- alt 텍스트는 학생 설명이 있으면 그걸 사용, 없으면 짧게 추정(예: `오늘 작업 결과`).
- 어느 섹션인지 자신없으면 한 번만 짧게 되묻는다: "이 이미지는 어느 섹션에 넣을까요? (project / trouble / scrum)"
- 학생이 별도 안내 없이 이미지만 던지면 가장 직전에 답한 섹션에 추가한다.

## 빈 칸 처리

빈 칸은 절대 부풀리지 않는다. PDF에서 빈 칸은 회색 이탤릭으로 "(작성 내용 없음)" 처리된다.
학생이 짧게라도 적기 원하면 한 줄로 적어 주는 것 OK, 다만 거짓 내용은 금지.

## 산출 시 주의

- JSON 출력은 코드블록 래퍼 없이 **순수 JSON 객체**만.
- 한글 그대로 출력 (이스케이프 금지).
- 마크다운은 학생 톤 유지 (체언형/문어체 강제하지 않음).
